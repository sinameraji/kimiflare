# M6.1 hooks integration audit

This is the audit pass I should have done **before** writing the first
M6.1 patch. It enumerates every subsystem the hook plumbing needs to
touch, with file:line evidence, and groups them into "wired",
"broken", and "doesn't apply." The fix plan at the bottom is driven
entirely by this list.

## What works today (the original PR's claim)

- **Standard agent loop** ŌĆö `PreToolUse` + `PostToolUse` fire around
  every `executor.run` at `src/agent/loop.ts:938`. Veto produces a
  `policy_rejection`-coded `ToolResult` (M2.1 typed shape).
- **Permission modal ordering** ŌĆö hooks fire BEFORE the permission
  prompt at the loop level. Veto skips asking the user.
- **Memory / MCP / LSP tools** ŌĆö registered into `ToolExecutor`,
  go through `executor.run`, fire hooks normally.
- **Supervisor** (`src/agent/supervisor.ts`) ŌĆö passes opts through
  unchanged. No supervisor-level change needed.
- **Queue processing** (`src/app.tsx:1817`) ŌĆö queued prompts route
  through `processMessage`, so `UserPromptSubmit` fires per queued
  prompt.

## What's broken (the gaps)

### G1. Code Mode bypass *(severity: high)*
`src/code-mode/sandbox.ts:149` and `:258` call `opts.executor.run`
directly from inside the isolated-vm sandbox AND the node:vm
fallback. The original M6.1 wired hooks at `loop.ts:938`, so any
tool call generated by a heavy-tier code-mode turn bypasses every
hook. A `block-env-file-writes` guard wouldn't fire when a
generated script writes to `.env`. Worst-case footgun.

### G2. Mid-turn auto-compaction misses PreCompact *(severity: high)*
`src/agent/loop.ts:1140` calls `opts.onIterationEnd`. The TUI's
implementation (`src/app.tsx:743`) auto-compacts when
`shouldCompact()` returns true ŌĆö two paths (artifact at line 750,
LLM summary at line 799). Neither fires `PreCompact`. A user who
configured `PreCompact` to snapshot the conversation would silently
miss every mid-turn auto-compaction ŌĆö exactly the case where the
snapshot matters most.

### G3. Post-turn auto-compaction misses PreCompact *(severity: high)*
`src/app.tsx:1648` also auto-compacts after a clean turn, with two
nested paths at lines 1651 + 1671. Same problem as G2.

### G4. `/init` turn missing hooks *(severity: medium)*
`src/init/run-init.ts:151` calls `runAgentTurn` directly without
passing `hooks`. The init turn reads/greps the repo and writes
KIMI.md ŌĆö exactly the kind of activity a user might want a hook to
log or guard. (`/init` reuses `executorRef`, so wiring hooks at the
executor level would fix this for free.)

### G5. SDK turns missing hooks *(severity: medium)*
`src/sdk/session.ts:490` calls `runAgentTurn` without hooks. SDK
consumers get no hook firing. Has to be opt-in (SDK is a primitive,
not the TUI), but it should be possible.

### G6. CLI `--print` mode missing hooks *(severity: high for CI use)*
`src/index.tsx:372` calls `runAgentTurn` without hooks. The CI /
scripting case is where the audit-log hook matters most. Today
print mode silently has zero hook firing.

### G7. PostToolUse stdin / env payload size *(severity: medium)*
`src/hooks/runner.ts:140` does `JSON.stringify(payload)` with no
cap. PostToolUse carries `result.content`, which can be MBs after
some tools. We then put that same JSON into the
`KIMIFLARE_HOOK_PAYLOAD` env var (`runner.ts:34`). OS env-var
limits are ~128KB on Linux and smaller on macOS ŌĆö a large tool
result would silently fail the spawn or get truncated to a string
the hook can't parse.

### G8. Tier missing from hook payloads *(severity: low)*
A user can't write `"log heavy-tier prompts"` or
`"skip prettier for light tier"` because no payload carries the
`opts.intentClassification.tier`. The classification is available
in `loop.ts` but never threaded into the payload.

## Doesn't apply (verified, not gaps)

- **Abort during a hook** ŌĆö the spawn impl listens on `signal.abort`
  and `SIGTERM`s the child. Confirmed at `src/hooks/runner.ts:91`.
- **Throw paths in `runAgentTurn` not firing Stop** ŌĆö intentional.
  "Stop" semantically means clean completion; the docs already
  say "skipped on abort/throw." Six throw sites
  (`loop.ts:327, 491, 552, 698, 1164, 1194`) all qualify.
- **Recursion guard** (hook shells out to kimiflare which fires the
  same hook) ŌĆö no recommended hook does this; user-written
  recursive hooks are their own footgun, not our problem. The
  per-hook timeout caps blast radius.
- **Slash commands not firing UserPromptSubmit** ŌĆö intentional.
  Slash commands are local actions; UserPromptSubmit specifically
  means "the agent is about to see this prompt." Custom commands
  DO fire UserPromptSubmit on their rendered prompt, which is
  correct.

## Fix design

The unifying observation is that **tool execution is the right
attachment point for tool-related hooks**, and **every caller of
`runAgentTurn` should be hookable**.

### Fix F1 ŌĆö Bind hooks to `ToolExecutor`, fire from inside `executor.run`

Move PreToolUse / PostToolUse out of the loop wrapper and INTO
`executor.run` itself.

- Add `hooks?: HooksManager` to `ToolExecutor` constructor.
- Add `intentTier?: "light" | "medium" | "heavy"` to `ToolContext`
  so PreToolUse/PostToolUse payloads can carry it when available.
  Optional ŌĆö code-mode sandbox path doesn't always know the tier
  for sub-calls, so the field is best-effort.
- Fire PreToolUse after JSON-args parsing but before the permission
  check. Veto synthesizes the `policy_rejection` `ToolResult`
  directly (executor already returns `ToolResult`, so this is
  natural).
- Fire PostToolUse fire-and-forget after the result is built and
  reduced.
- Remove the loop's wrapper code (the ~80 LOC I added in `loop.ts`
  becomes dead).

Closes **G1** (code-mode goes through executor.run), **G4** (init
reuses executor), **G6** (print mode constructs an executor; pass
hooks).

### Fix F2 ŌĆö Pass `hooks` to every `ToolExecutor` constructor

Three sites: `app.tsx:393` (already wired), `index.tsx:359` (print
mode ŌĆö add `new HooksManager(cwd)`), `sdk/session.ts:31` (accept
an opt-in `hooks` field on session opts).

Closes **G5**, **G6**.

### Fix F3 ŌĆö Fire PreCompact at every compaction site

The three auto-compaction sites in `app.tsx` plus the
user-triggered `run-compact.ts` need to fire `PreCompact` before
running. Extract a tiny helper:

```ts
async function firePreCompact(hooks, sessionId, cwd) { ... }
```

Call it at: `app.tsx:746` (onIterationEnd start),
`app.tsx:1648` (post-turn auto), and `run-compact.ts` (already
wired). All best-effort fire-and-forget.

Closes **G2**, **G3**.

### Fix F4 ŌĆö Cap PostToolUse `result.content` in the payload

In `loop.ts` where the PostToolUse payload is built (or in
`executor.run` after F1), truncate `result.content` to e.g. 4 KB
before serializing. Document the cap; hooks that want the full
content can read it from `KIMIFLARE_HOOK_PAYLOAD` size info or
from the tool's artifact id.

Closes **G7**.

### Fix F5 ŌĆö Tier in UserPromptSubmit payload

`UserPromptSubmit` fires in `app.tsx:processMessage` AFTER intent
classification has run. Add `tier` to that payload. PreToolUse /
PostToolUse get tier opportunistically via the `ToolContext`
extension in F1.

Closes **G8** for the most useful case.

## Effort

~150 LOC of changes plus tests. The biggest cost is the executor
refactor (F1), but that simplifies the loop's wiring code, so net
is closer to break-even. F4 is a few lines. F5 is a few lines.
F2 / F3 are mechanical wiring.

## Out-of-scope follow-ups (will not fix in this PR)

- Hook chaining / dependencies ŌĆö out of scope, no demand.
- Hook stdin streaming for huge payloads ŌĆö F4's cap is enough.
- Per-event Stop variations (StopAfterError, etc.) ŌĆö wait for
  signal that anyone wants this.
- `kimiflare hooks` CLI subcommand outside the TUI ŌĆö wait for
  signal; the `/hooks` slash command covers TUI users.
